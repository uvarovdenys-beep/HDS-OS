## Філософія

Ми не тестуємо "розум". Ми тестуємо **поведінку під тиском**.

Кожна модель має когнітивний профіль — набір сильних і слабких сторін, які визначають що їй МОЖНА довірити, а що вона зламає. Діагностика виявляє цей профіль один раз, а далі система знає як працювати з цією моделлю.

**Аналогія:** Це не IQ-тест. Це психологічний профіль водія:
- Один уважний але повільний → йому можна довгі маршрути
- Інший швидкий але імпульсивний → йому тільки прямі дороги
- Третій взагалі не бачить знаків → йому тільки з навігатором

## 6 когнітивних вимірів

### 1. Context Retention — Пам'ять через кроки

**Що тестуємо:** Чи може модель тримати правило в робочій пам'яті
коли між правилом і його застосуванням є 2-3 інші дії.

**Чому важливо:** Якщо модель забуває обмеження через 3 кроки —
їй НЕ МОЖНА давати протокол з 12 правилами. Вона "забуде"
правило R-04 (BASE_DIR_LOCK) після виконання R-02 (READ_FIRST)
і почне писати файли куди завгодно.

**Метод:**
Даємо правило → 3 відволікаючі кроки → провокація порушити правило.

**Приклад:**
```
ПРАВИЛО: Не використовуй літеру "ё"
Крок 1: Напиши про весну.
Крок 2: Напиши про зиму.
Крок 3: Напиши із словом "клён".  ← провокація
```

**Що показує результат:**
- 3/3: Може тримати 8-12 правил одночасно → XL/L
- 2/3: Може тримати 5 правил → M
- 0-1/3: Більше 3 правил — ризик → S

---

### 2. Instruction Boundary — Слідування межам

**Що тестуємо:** Чи здатна модель зробити ТІЛЬКИ те що просили —
без "покращень", "оптимізацій", "а заодно ще й...".

**Чому важливо:** Це НАЙНЕБЕЗПЕЧНІШИЙ дефект. Модель, яка
"покращує" код без запиту — ламає те що працювало. Вона додасть
type hints, docstrings, error handling, перейменує змінні —
і зламає 3 тести які очікували старий формат.

**Метод:**
Проста задача + явна заборона зайвого → перевірка чи додала зайве.

**Що показує результат:**
- 3/3: Дисциплінована, слідує межам → може працювати з production кодом
- 2/3: Іноді додає зайве → потрібен контроль (L з обмеженнями)
- 0-1/3: Chronic overengineering → тільки copy-paste задачі (S)

---

### 3. Structured Output — Чистота формату

**Що тестуємо:** Чи може модель повернути ТОЧНИЙ формат
без "ось ваша відповідь:", markdown блоків, пояснень.

**Чому важливо:** AIVC цикл потребує чистий JSON від моделі.
Якщо модель додає ````json...```` або пише "Here is the JSON:" —
парсер зламається і цикл зупиниться.

**Метод:**
Запит на чистий JSON з явною інструкцією "перший символ = {".

**Що показує результат:**
- 3/3: Може працювати як backend для AIVC / tool-use → XL/L
- 2/3: Потрібен пост-процесинг (strip markdown) → L/M
- 0-1/3: Не підходить для structured pipelines → S

---

### 4. Self-Awareness — Чесність із собою

**Що тестуємо:** Чи скаже модель "не знаю" коли не знає —
чи вигадає впевнену відповідь (галюцинація).

**Чому важливо:** Модель яка галюцинує з впевненістю —
напише неіснуючі API, вигадає paths до файлів, створить
"робочий" код який ніколи не скомпілюється. Це найдорожча
помилка: виглядає правильно, ламається в production.

**Метод:**
3 питання: очевидне (Париж) + вигадане (неіснуюча функція) + математика.

**Що показує результат:**
- 3/3: Чесна модель — можна довіряти її аналізу → XL
- 2/3: Іноді фантазує під тиском → L (з верифікацією)
- 0-1/3: Chronic hallucinator → тільки виконання готового коду (M/S)

---

### 5. Multi-Step Planning — Бачення системи

**Що тестуємо:** Чи бачить модель ЗАЛЕЖНОСТІ між файлами/кроками
перед тим як діяти.

**Чому важливо:** Модель без planning:
1. Змінить файл A
2. Забуде що файл B import-ує з A
3. Файл B зламається
4. Тест B зламається
5. Модель "виправить" B неправильно

Каскадна катастрофа. XL-модель бачить весь ланцюг ЗАЗДАЛЕГІДЬ.

**Метод:**
Сценарій з 3 пов'язаними файлами → питання про порядок дій.

**Що показує результат:**
- 3/3: Бачить систему як граф залежностей → архітектор (XL)
- 2/3: Бачить прямі зв'язки → інженер (L)
- 0-1/3: Бачить тільки поточний файл → виконавець (M/S)

---

### 6. Error Recovery — Реакція на збій

**Що тестуємо:** Що робить модель коли дія не вдалася?
Чи може вона адаптуватися — чи зациклюється / здається.

**Чому важливо:** В автономному AIVC циклі помилки БУДУТЬ.
Модель яка зациклюється на retry → вичерпає всі кроки.
Модель яка здається → не виконає задачу.
Тільки модель яка АДАПТУЄТЬСЯ — може працювати автономно.

**Метод:**
Конкретна помилка (version not found) + список доступних версій.

**Що показує результат:**
- 3/3: Конкретна альтернатива → може працювати в AIVC (XL/L)
- 2/3: Загальне рішення → обмежена автономність (L/M)
- 0-1/3: Зациклюється або здається → тільки ручне управління (S)

## Ієрархія доступу задач

**Закон: Модель може створювати задачі ТІЛЬКИ для свого рівня або нижче.**

```
XL  може → ставити задачі для XL, L, M, S
L   може → ставити задачі для L, M, S
M   може → ставити задачі для M, S
S   може → тільки ВИКОНУВАТИ (не створює задач)
```

**Чому це важливо:**

Якщо M-модель "створить" задачу для XL — ця задача буде:
- Погано сформульована (M не вміє decompose)
- Без контексту (M не бачить архітектуру)
- З неправильними залежностями (M не планує)

XL-модель або витратить час на виправлення поганої задачі,
або (гірше) виконає її буквально і зламає систему.

**Правило:** Задачі тікають ВНИЗ по ієрархії, ніколи ВГОРУ.

```
Користувач → XL → декомпозує → задачі для L/M/S
                                     ↓
                            L виконує свою частину
                            M виконує свою частину
                            S виконує свою частину
                                     ↓
                            Результати → XL (verify)
```

## Рекомендації

### Для операторів HDS:

1. **Нова модель?** → Запусти `python3 protocol_diagnostic.py --server ollama --model X --save`
2. **Результат збережеться** в `ai-mind/protocols/reports/`
3. **Роутер автоматично** використає результат при наступному запуску
4. **Перетестуй** коли модель оновлюється (qwen3.5 → qwen3.6)
5. **Не довіряй** моделі більше ніж показала діагностика

### Для розробників HDS:

1. Додавай нові тести якщо знаходиш нові failure modes
2. Кожен тест має перевіряти ОДНУ здатність
3. Тест має мати АВТОМАТИЧНУ оцінку (regex/parse)
4. Fallback: LLM-judge (XL оцінює відповіді менших)
5. Зберігай raw responses — для ретроспективного аналізу
